# CudaPipeline v2.0 - 架构设计文档

## 概述

CudaPipeline 是一个企业级的 GPU 计算调度和资源管理框架，专为多租户 AI 推理平台设计。核心目标是实现资源隔离、QoS 保障和最大化 GPU 利用率。

---

## 核心模块设计

### 1. Harness 框架层

**核心职责:**
- Skill (计算任务) 的生命周期管理
- 资源的动态分配与回收
- 错误隔离和故障恢复
- 监控和指标收集

**关键组件:**
- `SkillHarness`: 统一的执行入口
- `vGPUContext`: 虚拟 GPU 上下文
- 超时和重试机制

---

### 2. 资源虚拟化层

**设计原则:** 每个 Skill 看到的都是一个独立的虚拟 GPU

**vGPU 上下文包含:**
- 独立的显存配额
- 独立的 CUDA Stream 池
- 独立的 Kernel 执行配额
- 独立的超时和故障计数

**隔离边界:**
```
[ Skill A ] -- [ vGPU ctx A ] --+
                                   |
[ Skill B ] -- [ vGPU ctx B ] ----+-- [ 物理 GPU ]
                                   |
[ Skill C ] -- [ vGPU ctx C ] --+
```

---

### 3. 资源管理层

#### 3.1 配额管理器 (`ResourceQuotaManager`)

**核心特性:**
- 单例模式，全局唯一
- 多维度资源配额控制
- 硬限制 + 软限制
- 超限回调机制

**支持的资源类型:**
```cpp
enum class ResourceType {
    GPU_MEMORY,        // GPU 显存
    CPU_MEMORY,        // CPU 内存
    CUDA_STREAMS,      // CUDA Stream 数量
    CUDA_KERNELS,      // 并发 Kernel 数量
    THREADS,           // 线程数
    EXECUTION_TIME,    // 执行时间
    IO_BANDWIDTH,      // IO 带宽
};
```

**配额配置:**
```cpp
struct ResourceQuota {
    size_t hard_limit;  // 硬限制 - 超过直接拒绝
    size_t soft_limit;  // 软限制 - 超过警告但允许
    bool enabled;       // 是否启用
};
```

#### 3.2 令牌桶限流 (`TokenBucket`)

**用途:** 防止单个 Skill 消耗过多资源

**算法:**
- 桶容量 = 突发流量上限
- 填充速率 = 平均流量上限
- 支持阻塞和非阻塞模式

```
          Token Bucket
        ┌───────────────┐
Fill ──►    ○ ○ ○ ○    │ ──► Consume
        │  ○ ○        Capacity
        └───────────────┘
         Rate: N tokens/s
```

#### 3.3 RAII 资源分配器 (`ResourceAllocation`)

**自动资源管理:**
- 构造时分配资源
- 析构时自动释放
- 异常安全保证

---

### 4. 调度器层

#### 四种调度策略

| 调度器 | 适用场景 | 核心思想 |
|--------|----------|----------|
| **FCFS** | 简单流量 | 先来先服务，公平调度 |
| **Priority** | 多优先级 | 按优先级加权分配 |
| **EDF** | 实时任务 | 截止时间最早优先 |
| **RL** | 所有场景 | 强化学习自适应优化 |

#### RL 调度器架构

```
┌─────────────────────────────────────────┐
│  当前状态 (State)                      │
│  ├─ 队列长度                           │
│  ├─ 平均等待时间                       │
│  └─ 高优先级比例                       │
├─────────────────────────────────────────┤
│  动作空间 (Action)                      │
│  ├─ Batch Size: 1 / 2 / 4 / 8         │
├─────────────────────────────────────────┤
│  Q-Learning 算法                        │
│  ├─ ε-Greedy 探索策略                  │
│  ├─ 学习率衰减 α                      │
│  └─ 折扣因子 γ                         │
├─────────────────────────────────────────┤
│  奖励函数 (Reward)                      │
│  ├─ +10: 成功处理请求                  │
│  └─ +5: Batch 效率优化                 │
└─────────────────────────────────────────┘
```

### 5. 运行时层

#### 后端抽象

```cpp
class Backend {
    virtual bool initialize() = 0;
```

**支持的后端:**

## 关键设计决策

### 1. 资源隔离 vs 利用率权衡

**问题:** 严格隔离会降低资源利用率

**解决方案:**
```
低负载 → 允许资源借用 → 提高利用率
高负载 → 严格隔离保障 → 保证 QoS
```

### 2. 调度算法选择

**设计原则:**
- RL 调度器作为默认
- 其他调度器作为 fallback
- 支持运行时动态切换

### 3. 性能开销控制

**目标:** 框架开销 < 1%

**实现:**
- 无锁队列 (待实现)
- 批量操作减少同步
- 原子计数器替代锁

**实测:**
```
配额检查: 0.39 us/op
调度吞吐量: 3,496,503 req/s
```

## 线程安全模型

### 保证线程安全的组件

| 组件 | 线程安全方式 |
|------|-------------|
| ResourceQuotaManager | Mutex 保护 |
| TokenBucket | Mutex 保护 |
| RequestQueue | Mutex + Condition Variable |
| Metrics | 原子计数器 |

### 单线程组件

- vGPUContext (每个 Skill 独有)
- 调度器 (每个 Worker 独有)

## 扩展性设计

### 插件化架构

**扩展点:**
1. 新的调度器: 继承 `Scheduler` 基类
2. 新的后端: 继承 `Backend` 基类
3. 新的资源类型: 添加 `ResourceType` 枚举
4. 自定义监控: 注册回调函数

### 配置驱动

所有关键参数可配置:
```cpp
RLSchedulerConfig {
    double learning_rate;
    double discount_factor;
    double exploration_rate;
    int max_batch_size;
}
```
